# Пояснительная записка по проекту «Модуль базового фрод-мониторинга»

## Часть 1. Аналитическо-сравнительный анализ
В первой части работы была разработана сравнительная таблица, систематизирующая ключевые различия между классическими IT-проектами (например, CRM) и финтех-проектами (мобильные банки, P2P-платформы).

**Ключевые отличия финтех-проектов:**
* **Регуляторное соответствие:** Строгие требования центральных банков, соблюдение норм AML/KYC и международных стандартов безопасности.
* **Управление рисками:** Высокая концентрация на финансовых, операционных и репутационных рисках.
* **Безопасность:** Повышенные требования к защите персональных данных и транзакций.
* **Толерантность к ошибкам:** Критически низкая, так как любой сбой может привести к прямой потере денежных средств клиентов.
* **Команда:** Необходимость привлечения профильных экспертов — финансистов и юристов.

**Вывод:** Современные вызовы, такие как рост киберугроз и ужесточение регуляторики, делают разработку финансовых систем значительно более сложной и ответственной по сравнению с обычным ПО.

## Часть 2. Практическая реализация (Python)
В рамках практической части реализован Python-скрипт, моделирующий упрощенную систему фрод-мониторинга для анализа банковских операций.

### 1. Алгоритм и бизнес-правила
Система проверяет каждую транзакцию по трем основным правилам:
* **Контроль суммы (Rule 1):** Флаг подозрительности ставится, если сумма операции превышает 10 000 у.е.
* **Частота операций (Rule 2):** Система отслеживает активность пользователя. Если за последний час совершено более 5 транзакций, активность признается аномальной.
* **Регуляторное требование (География):** Для стран с повышенным риском (Кипр, Нигерия, Панама) допустимый лимит суммы автоматически сокращается в 2 раза.

### 2. Техническая спецификация
* **Стек:** Python 3.
* **Библиотеки:** `pandas` (для формирования отчетов и экспорта в Excel), `datetime` (для анализа временных окон транзакций).
* **Логика:** Использование скользящего временного окна (60 минут) для подсчета частоты операций.

### 3. Результаты
Программа генерирует отчет со следующими статусами:
* `APPROVED` — операция прошла проверку.
* `FLAGGED: High Amount` — превышен лимит суммы.
* `FLAGGED: High Frequency` — обнаружена слишком высокая частота платежей.
* `BLOCKED: High Risk Country & Amount` — автоматическая блокировка при рискованном сочетании страны и суммы.

Результат работы сохраняется в файл `fraud_report.xlsx`.

Разработана проектная документация для модуля фрод-мониторинга, включающая:
* **Цели и стейкхолдеры:** Определение ключевых ролей (комплаенс-офицер, риск-менеджер) и их требований к системе.
* **User Stories:** Описание функционала с точки зрения конечных пользователей системы мониторинга.
* **Митигация рисков:** * Для снижения числа ложных срабатываний внедрен статус «Проверка» (`FLAGGED`) вместо полной блокировки.
    * Для предотвращения дробления сумм внедрено правило контроля частоты.
    * Для минимизации задержек проведена оптимизация алгоритмов обработки данных.

3. Система создаст отчет `fraud_report.xlsx` с результатами анализа тестовой выборки.
